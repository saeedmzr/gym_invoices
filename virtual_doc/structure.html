System Design and Structure

Database

This product contains 5 tables :

  • 1 - users
  • 2 - invoices
  • 3 - clubs
  • 4 - checkins
  • 5 - memberships


users table is where we store and fetch users' information. We also fetch admin information from here. since its small a scale product and time-limited admin we authorize admin with only checking `is_admin` property in the users table.

clubs table is where we store the club's information. it contains club's name,cost_per_check_in and check_in_count.

memberships table is where we store user memberships of clubs. it contains user_id,club_id start_at,expire_at,status and credits . With every successful membership check-in, credits decrease by one.

invoices table contains invoice's month information of a user for a certain club.

checkins table contains invoice check-ins of a user for a certain club.



Repository Pattern

I tried to use laravel with best practice.

I used request,resource,models,routes just as laravel best practices are .

The only thing that might be challenging to know as you read code is Repositories classes.

I used repository pattern in this project and I create repository for each table.Also, i created a base repository to avoid the DRY principle.
